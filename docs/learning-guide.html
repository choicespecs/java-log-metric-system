Data Journeys

Journey 1 — Structured Metrics: Ingestion to TimescaleDB + Parquet

A client POSTs a MetricEvent JSON payload. The event traverses enrichment, windowed aggregation in Flink, and fans out to TimescaleDB for low-latency queries and MinIO Parquet for long-term analytics.

1
Client sends MetricEvent to metrics-collector :8081
The REST endpoint validates @NotBlank / @NotNull constraints via Spring Bean Validation. On failure, 400 is returned immediately without publishing.
POST /api/v1/metrics {"metricName":"cpu.usage","value":75.5,"unit":"percent", "serviceId":"svc-api","timestamp":"2024-01-15T10:30:00Z"}
2
MetricsPublisherService publishes to metrics.structured
Serialized as JSON string. Kafka producer uses acks=all, idempotence enabled, linger.ms=5, batch.size=65536. Key is serviceId to preserve per-service ordering.
produce(key="svc-api", value="{\"metricName\":\"cpu.usage\",...}")
3
StructuredMetricsConsumer (metrics-processor) enriches the event
Calls EnrichmentService.resolveMetadata(@Cacheable via Caffeine, max 500 entries, 10-min TTL). Looks up service_metadata in PostgreSQL to attach serviceName, team, environment, region.
SELECT * FROM service_metadata WHERE service_id = 'svc-api' -- Falls back to defaults if no row found
4
NormalizedMetric published to metrics.normalized
Includes all original fields plus enrichment: serviceName, team, environment, region, timestampMs (epoch ms for Flink watermarks).
5
Flink Pipeline 1 consumes metrics.normalized
Assigns event-time watermarks (5-second bounded out-of-orderness). Connects with CDC broadcast state for live enrichment overrides. Keys by serviceId:metricName and applies 1-minute TumblingEventTimeWindows.
WatermarkStrategy.forBoundedOutOfOrderness(Duration.ofSeconds(5)) .withTimestampAssigner((m, ts) -> m.getTimestampMs()) keyBy(m -> m.getServiceId() + ":" + m.getMetricName()) window(TumblingEventTimeWindows.of(Time.minutes(1)))
6
MetricWindowAggregator produces AggregatedMetric
Computes sum, min (initialized to Double.MAX_VALUE), max (initialized to Double.MIN_VALUE), count, and avg per window. MetricWindowProcessFunction attaches windowStart and windowEnd milliseconds.
7
Fan-out: TimescaleDbSink + Parquet FileSink
TimescaleDbSink issues UPSERT ON CONFLICT (metric_name, service_id, window_start) to metric_aggregates hypertable. FileSink writes Avro-encoded Parquet to MinIO bucket metrics-parquet, partitioned by environment/date, rolled on every checkpoint.
-- TimescaleDB upsert INSERT INTO metric_aggregates (...) VALUES (...) ON CONFLICT (metric_name, service_id, window_start) DO UPDATE SET ... -- MinIO path s3://metrics-parquet/aggregated/production/2024-01-15/part-0-uuid.parquet

Journey 2 — Raw Log Archival: Ingestion to HDFS to Parquet

Raw unstructured log lines arrive via REST, pass through Kafka, are archived by Flink to HDFS as rolling text files, and are later batch-processed by Spark into structured Parquet.

1
Client sends raw log line to metrics-collector
LogIngestionController accepts text/plain or JSON string array. No parsing at ingest time — collector is intentionally thin.
POST /api/v1/logs?source=svc-api Content-Type: text/plain 2024-01-15T10:30:00.123Z [ERROR] [svc-api] Connection timed out
2
Published to logs.raw topic (key=source)
No transformation. The raw string is the Kafka value. Keying by source groups lines from the same producer on the same partition.
3
Flink Pipeline 2 consumes logs.raw (no watermarks)
DataStream of String. No event-time processing — this pipeline is purely file archival. Uses DateTimeBucketAssigner for processing-time partitioning.
4
FileSink writes rolling .txt files to HDFS
Rolls every 5 minutes OR 128 MB OR 1 minute inactivity. Files are partitioned by hour so Spark processes bounded batches.
hdfs://namenode:9000/metrics/logs/raw/ year=2024/month=01/day=15/hour=10/ logs-abc123.txt logs-def456.txt.inprogress
5
Spark SparkLogProcessor batch job (periodic)
Reads all .txt files recursively under input path. Applies regex: (\d{4}-\d{2}-\d{2}T[\d:.]+Z)\s+\[(\w+)]\s+\[([^\]]+)]\s+(.*). Non-matching lines are filtered out.
spark-submit --class com.metrics.spark.SparkLogProcessor \ target/metrics-spark-processor-1.0.0-SNAPSHOT.jar \ hdfs://namenode:9000/metrics/logs/raw \ hdfs://namenode:9000/metrics/logs/parquet
6
Parquet output partitioned by level/date
Columns: timestamp, level, service_id, message, raw_line, date, level. Written in SaveMode.Append to HDFS.
hdfs://namenode:9000/metrics/logs/parquet/ level=ERROR/date=2024-01-15/ part-00000-*.parquet

Journey 3 — CDC Enrichment: PostgreSQL Changes to Flink Broadcast State

Changes to service metadata in PostgreSQL are captured by Debezium, published to Kafka, and loaded into Flink broadcast state so windowed aggregations always use fresh enrichment data without waiting for Spring cache expiry.

1
PostgreSQL configured with wal_level=logical
Enables logical decoding for CDC. Max 5 replication slots and 5 WAL senders configured in docker-compose. Debezium connects via a replication slot.
2
Debezium captures row-level change events
Kafka Connect (Debezium 2.5) publishes INSERT/UPDATE/DELETE/snapshot events to cdc.public.service_metadata. Each event contains op field: c=create, u=update, d=delete, r=snapshot.
{"op":"u","serviceId":"svc-api","serviceName":"API Service", "team":"platform","environment":"production","region":"us-east-1"}
3
Flink processBroadcastElement updates MapState
CdcEnrichmentBroadcastFunction maintains MapStateDescriptor keyed by serviceId. On c/u/r: broadcastState.put(serviceId, cdcEvent). On d: broadcastState.remove(serviceId).
4
NormalizedMetric events enriched from broadcast state
On each processElement call, Flink looks up broadcastState.get(serviceId). If found, overwrites serviceName, team, environment, and region fields before passing to windowed aggregation.

Journey 4 — Protobuf User Events: Gateway to Flink via Schema Registry

The API Gateway encodes UserEvent Protobuf messages with Confluent wire format and publishes to users.events. Flink deserializes using a cached Schema Registry client — zero registry calls per message after the first lookup.

1
Client POSTs user event to metrics-api-gateway :8083
UserEventController receives JSON, passes to UserEventPublisherService which serializes to Protobuf using the shared UserEvent proto defined in metrics-proto module.
POST /api/v1/users/events {"userId":"user-123","eventType":"PURCHASE","serviceId":"svc-checkout"}
2
KafkaProtobufSerializer auto-registers schema + publishes
On first publish, the serializer registers user_event.proto with Schema Registry and receives a schema ID. Subsequent messages reuse the same ID. Wire format: 0x00 (magic) + 4-byte schema ID + Protobuf bytes.
// 5-byte Confluent header byte[0] = 0x00 // magic byte byte[1..4] = schemaId (big-endian int) byte[5..] = protobuf-encoded UserEvent
3
Flink ProtobufUserEventDeserializer reads and caches schema
Reads the 5-byte header, extracts schema ID. On cache miss, calls CachedSchemaRegistryClient to fetch ParsedSchema and stores in per-task-slot Map. All subsequent messages for the same schema ID skip the registry call entirely.
4
UserEvent logged via PrintSinkFunction
Current implementation logs events. Extension point for writing to a dedicated store, triggering alerts, or joining with metric streams.
[UserEvent] userId=user-123 type=PURCHASE service=svc-checkout env=production ts=1705318200000

Journey 5 — External API Data: Unknown JSON to HDFS to Parquet

External APIs with unknown, evolving JSON schemas are ingested without any schema assumption. Flink archives raw NDJSON to HDFS; ExternalDataProcessor Spark job infers schema at read time and produces normalized Parquet.

1
External API data posted to metrics-collector
ExternalDataController accepts arbitrary JSON. X-Source-API header identifies the data source. Batch endpoint accepts JSON arrays.
POST /api/v1/external/ingest X-Source-API: payments-api {"orderId":"ord-99","amount":49.99,"currency":"USD"}
2
ExternalDataPublisherService publishes to external.data.raw
Key is X-Source-API header value, value is the raw JSON string. No schema transformation.
3
Flink Pipeline 4 wraps each record with metadata
ExternalDataDeserializer captures the Kafka key (sourceApi) alongside the raw JSON value. Each HDFS line written as NDJSON with envelope fields.
{"_source_api":"payments-api","_received_at":1714435200000, "_data":{"orderId":"ord-99","amount":49.99,"currency":"USD"}}
4
NDJSON files land in partitioned HDFS paths
Partitioned by source=<api>/year/month/day/hour. Rolls every 5 minutes or 128 MB. Spark reads bounded hourly batches.
hdfs://namenode:9000/metrics/external/raw/ source=payments-api/year=2024/month=01/day=15/hour=10/ data-abc.json
5
ExternalDataProcessor Spark job — schema inference + transform
spark.read.json() automatically infers schema. Flattens _data.* fields, normalizes column names, adds _ingested_at, _record_hash, _date columns. Optional outputs: S3/MinIO and JDBC (PostgreSQL/TimescaleDB).
spark-submit --class com.metrics.spark.ExternalDataProcessor ... \ hdfs://namenode:9000/metrics/external/raw \ hdfs://namenode:9000/metrics/external/parquet \ --output-s3 s3a://metrics-bucket/external/parquet \ --output-jdbc jdbc:postgresql://localhost:5433/metricsdb ...

Kafka Topics & Event Streams
metrics.structured
Key: serviceId
Format: JSON String
Partitions: 3
Retention: 168h
Raw structured metric events ingested by metrics-collector. Published by MetricsPublisherService with acks=all, idempotence, linger.ms=5. Consumer group: metrics-processor. Keyed by serviceId to guarantee per-service ordering and avoid cross-partition shuffles in downstream Flink jobs.
metrics-collector → metrics.structured → metrics-processor
logs.raw
Key: source
Format: Plain text
Partitions: 3
Retention: 168h
Raw unstructured log lines. Two independent consumer groups: (1) log-processor group in metrics-processor attempts regex parse and publishes matching lines to metrics.normalized; (2) flink-metrics-processor-raw-logs group in Flink Pipeline 2 archives all lines to HDFS without any parsing. Non-matching lines in metrics-processor are silently dropped.
metrics-collector → logs.raw → metrics-processor (regex) + Flink P2 (archive)
metrics.normalized
Key: serviceId
Format: JSON String
Partitions: 3
Retention: 168h
Enriched metric events produced by metrics-processor. Contains all MetricEvent fields plus enrichment: serviceName, team, environment, region (from PostgreSQL service_metadata), timestampMs (epoch milliseconds for Flink event-time watermarks), parsedLevel and logMessage (for log-derived events). Consumer group: flink-metrics-processor (Pipeline 1).
metrics-processor → metrics.normalized → Flink Pipeline 1
cdc.public.service_metadata
Key: auto (Debezium)
Format: JSON (Debezium envelope)
Partitions: 3
Producer: Debezium 2.5
CDC change events from PostgreSQL service_metadata table via Debezium logical decoding. Operations: c (create), u (update), d (delete), r (read/snapshot). Consumed by Flink's CdcEnrichmentBroadcastFunction to maintain broadcast state keyed by serviceId. Enables sub-second metadata propagation without waiting for Caffeine cache expiry (10-min TTL in metrics-processor).
PostgreSQL WAL → Debezium → cdc.public.service_metadata → Flink broadcast state
users.events
Key: userId (auto)
Format: Protobuf + Confluent 5-byte header
Partitions: 3
Schema: Schema Registry :8081
User behavioral events encoded as Protobuf with Confluent wire format (magic byte 0x00 + 4-byte schema ID + Protobuf payload). Published by metrics-api-gateway UserEventPublisherService via KafkaProtobufSerializer which auto-registers user_event.proto on first publish. Consumed by Flink Pipeline 3 via ProtobufUserEventDeserializer with per-task-slot schema cache.
metrics-api-gateway → users.events → Flink Pipeline 3
external.data.raw
Key: X-Source-API header
Format: Raw JSON string
Partitions: 3
Schema: None (schema-on-read)
Arbitrary external API JSON with no schema enforcement. Published by ExternalDataPublisherService in metrics-collector keyed by X-Source-API header. ExternalDataDeserializer in Flink captures the Kafka key (sourceApi) alongside the raw JSON payload and writes wrapped NDJSON to HDFS for downstream ExternalDataProcessor Spark job with schema inference.
metrics-collector → external.data.raw → Flink Pipeline 4 → HDFS NDJSON
