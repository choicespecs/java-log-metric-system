## Warnings and Observations

1. **Port collision note (informational):** The `docker-compose.yml` comment correctly notes that HDFS NameNode and MinIO both bind port 9000 within their containers but this is not a host-level conflict because only MinIO's 9000 and HDFS's 9870 are mapped to the host. No documentation action needed.

2. **Flink watermark double-assignment (tech debt):** `FlinkMetricsJob.java` sets `WatermarkStrategy.noWatermarks()` on `fromSource` then immediately calls `assignTimestampsAndWatermarks(forBoundedOutOfOrderness(...))` on the resulting stream. This is correct but redundant. Captured in `docs/DESIGN.md` technical debt table (row 9).

3. **`MetricWindowAggregator` sentinel leak (tech debt):** `createAccumulator()` initialises `min=Double.MAX_VALUE` and `max=Double.MIN_VALUE`. On an empty window `getResult()` resets both to `0.0`, not `null`. Downstream consumers must check `count=0` to distinguish "no data" from "all values were zero". Captured in `docs/DESIGN.md` technical debt table (row 4).

4. **TimescaleDB DDL not automated:** The `CREATE TABLE` and `create_hypertable` statements are documented only in `TimescaleDbSink.java` Javadoc. They must be run manually before the Flink job is submitted. This is now prominently documented in both `CLAUDE.md` and `docs/DESIGN.md` tech debt table (row 8).

5. **Auto-commit risk (tech debt — newly added):** `KafkaConfig` in `metrics-processor` uses `ENABLE_AUTO_COMMIT_CONFIG=true` with a 1-second commit interval. Messages can be lost if the service crashes between Kafka offset commit and successful downstream publish to `metrics.normalized`. Added as tech debt row 10 in `docs/DESIGN.md` and as security issue #11 in `docs/SECURITY.md`.

6. **`metrics-spark-processor` test requires local Spark session:** `SparkLogProcessorTest` spins up `SparkSession.builder().master("local[1]")`. This requires ~2GB heap and may time out in constrained CI environments. A note was added to `README.md` test-running section.

7. **`logs.raw` dual-consumer pattern:** Both `metrics-processor` (group `log-processor`) and `metrics-flink-processor` (group `flink-metrics-processor-raw-logs`) consume `logs.raw` independently. This means a matching log line is processed twice — once for near-real-time metric extraction via Spring, and once for HDFS archival via Flink. This is intentional by design (Decision 1) and documented in `docs/DESIGN.md`.
